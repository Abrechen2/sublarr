# v0.31.0 Phase 2 — Safe Cleanup Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Remove false safety signals, fix unbounded memory growth, and make silent failures visible — zero behavioral changes, zero new features.

**Architecture:** Three independent, low-risk changes in escalating scope: (1) cap the waveform audio cache, (2) delete the `_db_lock` no-op that clutters every DB call site, (3) add logging to silent `except: pass` blocks in the two modules with test coverage. Batch state persistence (spec §2.3) is deferred — it requires schema migration and is not safe cleanup.

**Tech Stack:** Python stdlib (`collections.OrderedDict`), Flask/SQLAlchemy (already in use), `logging`.

---

## Scope Notes

**Deferred: §2.3 Batch State Persistence** — `wanted_batch_state`, `_batch_extract_state`, `_batch_probe_state`, `batch_state`, `_scan_state`, `_orphan_state` are 88 in-memory dict references across `routes/wanted.py`, `routes/translate.py`, `routes/cleanup.py`. Migrating these to the `jobs` DB table requires schema migration, route rewrites, and frontend contract changes. This warrants its own plan (Phase 2b) — not appropriate for "safe cleanup."

**Exception audit scope:** There are ~85 silent `except: pass` patterns across the codebase. This plan fixes only the ones inside `wanted_search.py` and `providers/__init__.py` — the two modules now covered by Phase 1 tests, so regressions will be caught immediately.

## Task 1: Cap the Waveform Cache

**Files:**
- Modify: `backend/routes/tools.py`

**Context:** `WAVEFORM_CACHE: dict = {}` on line 19 maps `(video_path, mtime)` tuples to temp `.opus` file paths. It grows indefinitely — each unique video visited adds an entry and its temp file is never evicted. Replace with a size-capped `OrderedDict` that evicts the oldest entry (and deletes its temp file) when the cap is exceeded.

- [ ] **Step 1.2: Replace the declaration (line 19)**

Change:
```python

```
To:
```python

_WAVEFORM_CACHE_MAX = 100
WAVEFORM_CACHE: OrderedDict = OrderedDict()  # (video_path, mtime) -> temp_opus_file_path
```

**Note:** `from collections import OrderedDict` goes at the top of the import block with the other stdlib imports, NOT inline at line 19. Move the constant and the OrderedDict declaration to line 19.

The actual placement:
1. Add `from collections import OrderedDict` to the stdlib imports near the top of the file (check existing imports at lines 1-15, add after `import tempfile` or wherever alphabetically fits).
2. Replace line 19 declaration with:
```python
_WAVEFORM_CACHE_MAX = 100
WAVEFORM_CACHE: OrderedDict = OrderedDict()  # (video_path, mtime) -> temp_opus_file_path
```

- [ ] **Step 1.3: Add eviction on cache write (around line 2614)**

Find the write:
```python
WAVEFORM_CACHE[cache_key] = tmp.name
```

Replace with:
```python
WAVEFORM_CACHE[cache_key] = tmp.name
# Evict oldest entry if over capacity, delete its orphaned temp file
if len(WAVEFORM_CACHE) > _WAVEFORM_CACHE_MAX:
    _, evicted_path = WAVEFORM_CACHE.popitem(last=False)
    try:
        if os.path.exists(evicted_path):
            os.remove(evicted_path)
    except OSError as e:
        logger.debug("Waveform cache eviction cleanup failed: %s", e)
```

## Task 3: Scoped Exception Audit

**Files:**
- Modify: `backend/wanted_search.py`
- Modify: `backend/providers/__init__.py`

**Context:** There are ~85 silent `except: pass` patterns in the codebase. This task fixes only the ones inside `wanted_search.py` and `providers/__init__.py` — the two modules with Phase 1 test coverage. The rule: **never swallow silently unless the fallback is explicitly documented AND the exception is bound with `as e`**. Acceptable outcomes:
1. `except (SpecificError1, SpecificError2) as e:` + `logger.debug(...)` — narrow type, logged
2. `except Exception as e:` + `logger.debug(...)` with a comment explaining why it's non-fatal

**Important:** Do NOT change behavior. Only add logging and narrow exception types where unambiguous. If narrowing the type requires understanding production code you haven't read, leave it as `Exception` and just add the logging.
